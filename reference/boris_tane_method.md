# Boris Tane 三步工作法

**來源**: Boris Tane 九個月 AI 協作開發實戰經驗

---

## Quick Reference

### 三步流程
1. **理解** → 產出 `research.md`，人類審核
2. **計劃** → 產出 `plan.md`，批注循環 1-6 次
3. **執行** → 嚴格按 plan 實作，標記進度

### 核心規則
- 每一步都加「**先不要寫代碼**」
- 計劃不滿意就不執行
- 大問題直接重做，不修補
- 計劃文件必須持久化（檔案，不是對話）

---

## 核心理念

> **「先不要寫代碼」是最難執行的指令**

AI 協作的最大風險不是語法錯誤，而是**方向錯誤**：
- 代碼能跑，但破壞了系統架構
- 忽略現有快取層
- 違反 ORM 規範
- 重複實作已有功能

**解決方案**：把「思考」和「執行」徹底分開。

---

## Step 1: 讓 AI「讀懂」再「寫下」

**目標**: 確保 AI 真正理解現有系統

**指令範例**:
```
請深入分析以下專案，理解其架構、技術棧、資料流、慣例，
並將你的理解寫成 research.md。

重點包括：
1. 專案結構與分層架構
2. 資料庫 Schema 與關聯
3. API 設計慣例
4. 使用的框架與套件
5. 現有的快取策略
6. 錯誤處理模式
7. 測試覆蓋情況

**先不要寫代碼**，只產出 research.md
```

**輸出**: `research.md`
- 真正讀者是**人類**
- 用來驗證 AI 是否理解了系統
- 誤解在這一步消滅，不等到代碼裡

**審核清單**:
- [ ] AI 是否理解核心架構？
- [ ] 是否注意到關鍵設計模式？
- [ ] 是否識別出潛在衝突點？
- [ ] 有沒有誤解現有功能？

如果發現誤解 → 立即糾正，要求重寫。

---

## Step 2: 計劃、批注、反覆修改

**目標**: 把人類的隱性判斷轉化為明確指令

**指令範例**:
```
基於 research.md 的理解，為 [任務描述] 生成詳細的實作計劃 plan.md。

計劃應包含：
1. 實作步驟清單（按順序）
2. 需要修改的檔案
3. 新增的檔案結構
4. API 設計
5. 資料庫變更（如有）
6. 測試策略
7. 潛在風險與應對

**先不要寫代碼**，等我審核計劃。
```

**批注循環** (1-6 次迭代):

1. 人類審核 plan.md
2. 直接在文件中加批注：
   ```markdown
   1. 建立 User model
      > 【批注】使用現有的 BaseModel，不要重新實作
   2. 實作 authentication middleware
      > 【批注】改用 Passport.js，不用自己寫
   3. 建立快取層
      > 【批注】不可選 - 效能要求不高，先跳過
   ```
3. 扔回 AI：「按批注更新 plan.md。**先不要寫代碼**。」
4. 重複直到滿意

**何時停止迭代**:
- ✅ 計劃符合所有業務約束
- ✅ 技術選型合理
- ✅ 沒有重複造輪子
- ✅ 考慮了邊界情況
- ✅ 測試策略清晰

**關鍵**: 每次都加「**先不要寫代碼**」— AI 一旦覺得「差不多了」就會衝動執行。

---

## Step 3: 一次性執行

**目標**: 創造性工作已完成，執行應該是無聊的

**指令範例**:
```
按照 plan.md 全部實現。

規則：
1. 嚴格按照計劃執行，不要自作主張
2. 完成一項標記一項 [x]
3. 不到結束不要停
4. 遇到問題記錄在 issues.md，繼續執行
5. 全部完成後報告
```

**糾錯**: 簡短明確
```
"你沒實現 deduplicateByTitle 函數。"
"這個設置頁應該在 admin 應用裡。"
```

**重做 vs 修補**: 出了大問題 → 直接回滾，縮小範圍重來。**重做成本幾乎總比修補低。**

---

## 為什麼這個方法有效

### 1. 持久化的計劃文件

**問題**: AI 上下文會退化（尤其長對話）
**解決**: plan.md 作為檔案存在，隨時可以重新指向它

```bash
# 即使 token 用盡，重新開始時
"繼續執行 plan.md 中未完成的項目"
```

### 2. 分離對齊與執行

**傳統錯誤流程**:
```
輸入需求 → AI 生成代碼 → 報錯 → 修改 → 再生成 → 再報錯 → ...
（上下文混亂，陷入死循環）
```

**正確流程**:
```
理解系統 → 寫下理解 → 審核理解 →
生成計劃 → 批注計劃 → 修改計劃（1-6次）→
確認計劃 → 一次性執行
```

### 3. AI 從「執行者」變成「被審稿的作者」

- `research.md` = 初稿（理解）
- `plan.md` = 修訂稿（計劃）
- 人類批注 = 紅筆

> **寫作的秘密從來不是下筆如神，是改稿改到吐。AI 寫代碼，同理。**

---

## ✅ DO

1. 強制分離理解、計劃、執行
2. 每次都說「先不要寫代碼」
3. 用白紙黑字驗證 AI 的理解
4. 直接在計劃文件中批注
5. 反覆修改計劃直到滿意（1-6 次）
6. 大問題直接重做，不修補
7. 保持計劃文件持久化

## ❌ DON'T

1. 讓 AI 邊想邊做
2. 口頭審核計劃（要寫下來）
3. 計劃不滿意就執行
4. 執行時改計劃（應該停下來重新規劃）
5. 上下文混亂時硬修補（應該重來）

---

> **AI 的價值不是「生成」，而是「對齊」。慢下來對齊，才是最快的路。**
