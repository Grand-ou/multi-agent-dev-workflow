# Boris Tane 方法論整合

**參考來源**: https://www.threads.com/@ad.life_engineer/post/DVQz_pPkWcc

---

## 核心理念

> **「先不要寫代碼」是最難執行的指令**

AI 協作開發的最大風險不是語法錯誤，而是**方向錯誤**：
- 代碼能跑，但破壞了系統架構
- 忽略現有快取層
- 違反 ORM 規範
- 重複實作已有功能

**解決方案**：把「思考」和「執行」徹底分開。

---

## 三步工作流

### Step 1: 讓 AI「讀懂」再「寫下」

**目標**: 確保 AI 真正理解現有系統

**操作**:
```bash
# 指令範例
"請深入分析以下專案，理解其架構、技術棧、資料流、慣例，
並將你的理解寫成 research.md。

重點包括：
1. 專案結構與分層架構
2. 資料庫 Schema 與關聯
3. API 設計慣例
4. 使用的框架與套件
5. 現有的快取策略
6. 錯誤處理模式
7. 測試覆蓋情況

**先不要寫代碼**，只產出 research.md"
```

**輸出**: `research.md`
- 這份文件的真正讀者是**人類**
- 用來驗證 AI 是否真的理解了系統
- 誤解在這一步消滅，不等到代碼裡

**審核重點**:
- [ ] AI 是否理解專案的核心架構？
- [ ] 是否注意到關鍵的設計模式？
- [ ] 是否識別出潛在的衝突點？
- [ ] 有沒有誤解現有功能？

**如果發現誤解**: 立即糾正，要求重寫 research.md

---

### Step 2: 計劃、批注、反覆修改

**目標**: 把人類的隱性判斷轉化為明確指令

**操作**:
```bash
# 指令範例
"基於 research.md 的理解，為 [任務描述] 生成詳細的實作計劃 plan.md。

計劃應包含：
1. 實作步驟清單（按順序）
2. 需要修改的檔案
3. 新增的檔案結構
4. API 設計
5. 資料庫變更（如有）
6. 測試策略
7. 潛在風險與應對

**先不要寫代碼**，等我審核計劃。"
```

**輸出**: `plan.md`（第一版）

**批注循環** (1-6 次迭代):
1. **人類審核** plan.md
2. **直接在文件中加批注**:
   ```markdown
   ## 實作步驟
   1. 建立 User model
      > 【批注】使用現有的 BaseModel，不要重新實作
   2. 實作 authentication middleware
      > 【批注】改用 Passport.js，不用自己寫
   3. 建立快取層
      > 【批注】不可選 - 效能要求不高，先跳過
   ```
3. **扔回給 AI**:
   ```bash
   "按批注更新 plan.md。先不要寫代碼。"
   ```
4. **重複**直到計劃滿意為止

**何時停止迭代**:
- ✅ 計劃符合所有業務約束
- ✅ 技術選型合理
- ✅ 沒有重複造輪子
- ✅ 考慮了邊界情況
- ✅ 測試策略清晰

**關鍵提醒**: 每次都加「**先不要寫代碼**」
> 因為 AI 一旦覺得「差不多了」就會衝動執行，而這個時機通常不對。

---

### Step 3: 一次性執行

**目標**: 創造性工作已完成，執行應該是無聊的

**操作**:
```bash
# 指令範例
"按照 plan.md 全部實現。

規則：
1. 嚴格按照計劃執行，不要自作主張
2. 完成一項標記一項 [x]
3. 不到結束不要停
4. 遇到問題記錄在 issues.md，繼續執行
5. 全部完成後報告"
```

**監控**:
- 定期檢查 plan.md 的完成度（✓ 標記）
- 如有問題立即指出（上下文還在，一句話夠）

**糾錯範例**:
```bash
# 簡短明確
"你沒實現 deduplicateByTitle 函數。"
"這個設置頁應該在 admin 應用裡。"
```

**重做 vs 修補**:
- 出了**大問題**？→ 直接**回滾**，縮小範圍重來
- 經驗：**重做成本幾乎總比修補低**

## 為什麼這個方法有效

### 1. 持久化的計劃文件

**問題**: AI 上下文會退化（尤其長對話）
**解決**: plan.md 作為檔案存在，隨時可以重新指向它

```bash
# 即使 token 用盡，重新開始時
"繼續執行 plan.md 中未完成的項目"
```

### 2. 分離對齊與執行

**傳統錯誤流程**:
```
輸入需求 → AI 生成代碼 → 報錯 → 修改 → 再生成 → 再報錯 → ...
（上下文混亂，陷入死循環）
```

**正確流程**:
```
理解系統 → 寫下理解 → 審核理解 →
生成計劃 → 批注計劃 → 修改計劃（1-6次）→
確認計劃 → 一次性執行
```

### 3. 把 AI 從「執行者」訓練成「被審稿的作者」

- `research.md` = 初稿（理解）
- `plan.md` = 修訂稿（計劃）
- 人類批注 = 紅筆

**寫作的秘密從來不是下筆如神，是改稿改到吐。AI 寫代碼，同理。**

## 整合到 Multi-Agent Workflow

### Phase 0+: 深度理解階段（新增）

在啟動任何 Sub-Agent 之前：

1. **主 Agent 先讓 AI 產出 `research.md`**
   ```bash
   # 針對整個專案
   sessions_spawn(
     task="深入分析 ai-persona-test 專案，產出 research.md",
     agentId="code-agent",
     label="research-phase",
     cleanup="keep"
   )
   ```

2. **人類審核 research.md**
   - 檢查 AI 是否理解架構
   - 糾正任何誤解
   - 標記重要的設計決策

3. **將 research.md 加入所有後續 Sub-Agent 的 context**

### Phase 1+: 計劃批注階段（強化）

對每個任務：

1. **Sub-Agent 先產出 `plan-[任務名].md`**
   ```bash
   # 指令範例
   "基於 research.md，為前端框架搭建任務生成 plan-frontend-setup.md。
   
   **先不要寫代碼**，等我審核。"
   ```

2. **人類批注循環**（1-6 次）
   - 直接在 plan 文件中加批注
   - 要求 AI 按批注更新
   - **每次都加**「先不要寫代碼」

3. **確認後才執行**
   ```bash
   "計劃已確認，開始執行 plan-frontend-setup.md。
   
   規則：嚴格按計劃、標記進度、不要停。"
   ```

### 文件結構（更新）

```
project-root/
├── research.md              # AI 對現有系統的理解（新增）
├── PLAN.md                  # 整體開發計劃
├── plan-task-1.md           # 任務 1 詳細計劃（新增）
├── plan-task-2.md           # 任務 2 詳細計劃（新增）
├── AGENTS_STATUS.md         # Sub-Agent 狀態追蹤
├── issues.md                # 執行中遇到的問題（新增）
└── INTEGRATION_REPORT.md   # 整合測試報告
```

## 實戰範例：應用到 IntradayLab

### 步驟 1: 理解現有系統

```bash
# 在 /var/www/fmg-knowledge
sessions_spawn(
  task="深入分析 IntradayLab Knowledge Base 專案：

分析目標：
- 現有 index.js 的架構
- PostgreSQL + pgvector 的使用方式
- 知識文章的資料結構
- RAG 查詢的實作方式
- OpenAI API 的整合模式
- Nginx 設定與部署架構

產出 research.md，描述你對這個專案的完整理解。

**先不要寫代碼**。",
  agentId="code-agent",
  label="research-intradaylab",
  cleanup="keep"
)
```

### 步驟 2: 審核理解

人類檢查 `research.md`：
- [ ] 是否理解 pgvector 的索引設定？
- [ ] 是否注意到相似度門檻 0.3？
- [ ] 是否理解 Google OAuth 的回調路徑？
- [ ] 有沒有誤解資料庫結構？

**如有誤解**: 直接在 research.md 加批注 → 要求重寫

### 步驟 3: 計劃第一個任務

```bash
sessions_send(
  sessionKey="agent:main:isolated:research-intradaylab",
  message="基於你的理解，為「前端框架搭建」任務生成 plan-frontend-setup.md。

要求：
1. 在 /var/www/fmg-knowledge/web/ 建立 Vite + React
2. 整合現有後端（index.js）
3. 設定 Nginx 正確 serve 前端
4. 考慮生產環境部署

**先不要寫代碼**，產出計劃後等我審核。"
)
```

### 步驟 4: 批注循環

**第一版 plan-frontend-setup.md**:
```markdown
## 實作步驟
1. 初始化 Vite 專案
2. 安裝 React + TypeScript
3. 安裝 Tailwind CSS
   > 【批注】使用 shadcn/ui，預設已含 Tailwind
4. 建立 API 呼叫層
   > 【批注】使用 Axios，並設定 baseURL 為 /api
5. 修改 Nginx 設定
   > 【批注】不可行 - Nginx 設定已存在，只需在現有基礎上加前端路由
...
```

**扔回去**:
```bash
"按批注更新 plan-frontend-setup.md。先不要寫代碼。"
```

**第二版審核** → **第三版** → ... → **確認！**

### 步驟 5: 執行

```bash
"計劃確認，開始執行 plan-frontend-setup.md。

規則：
1. 嚴格按計劃，不自作主張
2. 完成一項標記 [x]
3. 遇到問題記錄 issues.md，繼續
4. 全部完成報告

現在開始。"
```

## 關鍵原則總結

### ✅ DO

1. **強制分離理解、計劃、執行**
2. **每次都說「先不要寫代碼」**
3. **用白紙黑字驗證 AI 的理解**
4. **直接在計劃文件中批注**
5. **反覆修改計劃直到滿意（1-6 次）**
6. **大問題直接重做，不修補**
7. **保持計劃文件持久化**

### ❌ DON'T

1. ❌ 讓 AI 邊想邊做
2. ❌ 口頭審核計劃（要寫下來）
3. ❌ 計劃不滿意就執行
4. ❌ 執行時改計劃（應該停下來重新規劃）
5. ❌ 上下文混亂時硬修補（應該重來）

---

## 一個決策原則

> **AI 的價值不是「生成」，而是「對齊」**

- 代碼寫錯了能報錯
- 方向錯了沒人攔你

Boris 真正在做的，是把 AI 從「執行者」訓練成「被審稿的作者」。

**慢下來對齊，才是最快的路。**

---

## 結語

這套方法沒有魔法 Prompt，沒有複雜工具鏈。

本質就是一件事：
> **動手之前，把思路逼到紙面上，反覆修到滿意。**

Boris 沒說的是：
> **這需要你自己先把需求想清楚，AI 幫不了你。**
